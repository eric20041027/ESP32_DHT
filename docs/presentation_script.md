## 投影片 3｜[B] 挑戰：現有技術的極端兩難

**標題**：[B] 挑戰 — 現有技術的極端兩難

- ✅ 優勢：精準度極高、數據可靠

> 缺乏一個「**低功耗、高性價比且具備強大連網能力**」的破局點

### 消費級感測器
- ✅ 優勢：價格低廉、普及率高

**補充痛點**：固定式監控與感測器存在無法克服的「物理覆蓋死角」（如 15m/3m 極限），且僅能被動記錄，缺乏靈活應對動態事件的能力。

## 投影片 4｜[C] 解藥：開源物聯網的黃金三角

**標題**：[C] 解藥 — 開源物聯網的黃金三角

### 三角架構

**① 物理感知層（DHT22）**
- 低功耗、長期穩定的精準數位溫濕度特徵提取
- 微安培（μA）級極低功耗，邊際成本極低

**② 邊緣運算層（ESP32）**
- 強大微控制器，內建 Wi-Fi/Bluetooth 雙模
- 負責資料預處理、閾值判斷與封包路由

**③ 雲端應用層（IoT Cloud）**
- 突破空間限制，實現零時差遠端監控與長期儲存

**結論**：透過此架構，能以消費級的成本，實現工業級的資料監測擴展性。

### 技術診斷比較表

| 比較項目 | 傳統指針儀表 | 舊型 DHT11 | 本系統（DHT22 + ESP32） |
|---------|:---------:|:---------:|:---------------------:|
| 特徵提取精度 | 肉眼辨識 | ±2°C, ±5%RH | ✅ 0.1°C 精度，高靈敏 |
| 運算效力 | 無訊號 | 類比/低階數位 | ✅ 標準化數位訊號輸出 |
| 雲端連動 | 實體孤島 | 需外加繁雜模組 | ✅ 原生 Wi-Fi 即時串流 |
| 系統擴充性 | 零 | 低 | ✅ 無限擴充至智慧家居 |

## 投影片 6｜[D] 數據流動脈絡

**標題**：[D] 資訊架構 — 邊緣與雲端的雙向橋接

### 四層數據流
```
感測神經末梢  →  大腦運算核心  →  雲端神經網路        →  雙維度應用
   DHT22          ESP32        ThingSpeak + Blynk     室內 / 戶外
環境溫濕度採取  邊緣運算與封包化  數據收斂 & 互動迴路   智慧家居 / 無人機
```

### Step 1 — 邊緣採集
ESP32 以程式碼控制採集頻率，將數位訊號打包為標準化數據封包

### Step 2 — 雲端分流
- **ThingSpeak**：建立長期趨勢資料庫與數學模型驗證
- **Blynk**：打造低延遲的遠端監控儀表板

### Step 3 — 終端反應
使用者透過手機 App 即時讀取數據；異常觸發時，系統可反向發送控制指令至 ESP32

**結論**：本系統捨棄傳統定點框架，建立從微安培級感測到廣域巡航的完整閉環架構。

## 投影片 7｜硬體接線規格

**標題**：[D] 實作藍圖 — 硬體拓樸的極簡美學

### 接線對照表

| DHT22 腳位 | 連接至 ESP32 | 說明 |
|-----------|:-----------:|------|
| VCC (Pin 1) | 3.3V | 低功耗供電 |
| DATA (Pin 2) | GPIO4 | 單一數位資料線 + 10kΩ 上拉電阻 |
| GND (Pin 4) | GND | 接地回路 |

### 設計哲學
- 僅需 **3 條線 + 1 顆電阻**完成完整系統接線
- 極致降低硬體佈線複雜度，減少物理層除錯成本
- 將重心轉移至軟體與雲端架構

```
└── README.md
```

### 核心程式邏輯
```
setup()  →  Wi-Fi 連線 → Blynk 初始化 → ThingSpeak 初始化

loop()
  ├── GPIO0 = HIGH → 連續上傳模式（室內市電）
  └── GPIO0 = LOW  → 深度休眠模式（戶外電池）
      └── readAndUpload()
          ├── DHT22 讀取
          ├── 上傳 ThingSpeak
          ├── 更新 Blynk 儀表板
          └── 閾值判斷 → Push Notification
```

### 資安設計
- `secrets.h`（含真實憑證）→ 加入 `.gitignore`，**永不上傳 GitHub**
- `secrets.example.h`（佔位字串）→ 安全公開供參考
